Design Philosophy

Some background on what makes Ibis.js special and when to use it...

What is Ibis.js?

...and what makes it's approach different?

Ibis.js is a JavaScript mapping library designed for the efficient display of geospatial data. It differs in two key ways from most existing mapping libraries e.g. Leaflet or OpenLayers.

  • it doesn't use rasterised map tiles. Instead it draws entity GeoJSON data - directly in the browser itself via WebGL.
  • it doesn't load data in a lazy, "on-the-fly" manner. Instead, it loads eagerly - grabbing and assimiating MBs of geospatial data in seconds - once only, on load..

Why is another approach necessary?

...what are the pros and cons of images, tiles and vector data?

There is no "one size fits all" approach to mapping. Everything depends on the type of data and type of interaction wanted with that data. Image tile approaches work extremely well for high-level zoom overviews of complex data. You can pack a lot of geospatial graphical detail into one 256x256 pixel tile! In addition "lazy", "on-the-fly" loading strategies work well for casual, short user sessions. However, the situation changes radically, if we need to be working close-up (e.g. zoom levels 17+ by slippy-tile terms) at our geo data or if the application needs to handle heavy-usage with sessions measured in hours or even days - not minutes. The reason for this comes down to the type of map application and the type of user experience wanted. For casual users - a user who just wants to quickly look-up something on a map - the 'on-the-fly' approach works well enough. Load is fast and data usage is low. However, for professional users of geo-centric web applications where an individual might be working on an application all day- the situation is very different! Their experience is the constant drag of loading and reloading of geo tiles - and, unless some very smart backend handling has been added too - the complete amnesia of the map when an area previously worked on - gets reloaded 'afresh' Additionally, beyond user annoyance and with a bottom-line cost - is the serious network bandwidth consumption such 'on-the-fly' tile-based applications can cause host and webmaster. The raster tile equivlent of the Greater London demo included with Ibis - isn't a few megabytes - it's gigabytes! And of course, all that bandwidth costs money. So Ibis's different approach is not 'on-the-fly' load - but rather - it eagerly loads the ENTIRE dataset 'up-front'. This may seem counter-intuitive from typical web-wisdom - but in fact - for serious applications - it delivers a better OVERALL user experience. A single 5-15 second load - and then no more downloads, no more browser staggering and no more bandwidth usage full-stop.

What can it do?

...and when to use it?

  • display large multi-layered sets of GeoJson vector data (especially heavy datasets i.e. > 50Mb records).
  • Work both with online and offline browsing (see Road Map section below for latter).
  • Works across all desktop browsers (and some mobile devices).
  • Entire library minimised and gzipped is < 100Kb! About the cost of four 256x256 map tiles!
As I hope is already clear, if you're interested in a library/component for casual map browsing and mobile experiences... don't use this!... use the very light and fast Leaflet.js instead. How stable is it?

...and what are it's limits?

In terms of general library stability - it should be stable enough for most typical production senarios. More tests will be added to the library over time - thus improving that edge cases are correctly dealt with - but if you discover any faults whilst using Ibis.js - please report a bug. The one exception to this stabilty - is if you are trying to cram an ocean into a champagne glass... because it just won't work! Really!!! ;)) Browsers have their memory capacities and although this limit can be worked with - it can't be ignored. For example, in a Chrome process - around 200-250Mb of user data is about the (current Sep 2016) absolute limit before the browser explodes! Even approaching the limit's of its memory other unexpected behaviour i.e. sporadic freezing and process death can be expected! Thus, forewarned, a good rule of thumb is to work within two-thirds of the maximum capacity. Using that - with all main browsers - stable and consistently responsive map usage can be expected - even through long user-sessions.
